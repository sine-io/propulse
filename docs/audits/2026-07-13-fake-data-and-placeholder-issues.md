# 假数据与占位功能审计问题清单（2026-07-13）

## 1. 审计目的

逐项记录当前网页中会被用户误认为真实数据、真实判断或已完成功能的内容，方便后续按问题单逐一修复。

本清单不是“删除所有静态内容”：

- 首页可以保留演示场景，但每组演示数字和结论必须就近标注“示例”，不能让用户误认为是自己的数据。
- 判断方法可以保留静态文章，但目录、引用和适用条件必须真实可用。
- 测试夹具和 OpenAPI 示例不属于运行时假数据；生产组件引用的 `sample-data.ts`、接口失败时展示的样例、无行为按钮属于本次范围。
- 输入框里的提示语本身不是问题；页面声称可以保存、复制、提醒或导出，但实际没有对应行为才是问题。

## 2. 审计范围与方法

- 页面：`/`、`/calculator`、`/neighborhoods`、`/action-window`、`/methods`、`/watchlist`、`/templates`。
- 数据链路：前端 API client、Go HTTP handler、应用服务、指标计算、数据库 schema、定时任务和 Docker Compose。
- 方法：运行本地 Next.js 页面并确认全部路由可访问；逐页核对渲染内容、事件处理和 API 来源；反查后端数据生成过程。
- GitHub：审计时仓库没有已有 Issue。本次先记录到本地，下面每个三级标题均可直接转成一个独立 GitHub Issue。
- 现状：以下问题默认均为“待处理”。

## 3. 优先级定义

- `P0`：会把样例或不可靠推断伪装成真实个人/市场结论，可能直接影响购房决策，必须先处理。
- `P1`：关键流程或控件只是占位，或缺少真实数据闭环，应在主要页面可用前处理。
- `P2`：不会立即产生错误决策，但会制造误解、维护负担或体验落差。

## 5. 系统与数据链路问题

### SYS-001 [P0] 生产逻辑固定使用共享的 `demo-user`

**证据**

- `internal/interfaces/http/handler/watchlist.go` 固定声明 `demoUserID = "demo-user"`，测算和观察池请求都写入这个用户。
- `internal/application/decision/query.go` 总是读取该用户最近一次测算和观察池。
- `migrations/000001_initial_schema.up.sql` 给个人数据表设置同一个默认用户。
- 顶部“登录”并没有实际访问控制，见 NAV-001。

**问题与影响**

不同访问者会共享个人测算、观察池和出手建议。页面显示的“你的预算”和“我的观察池”无法证明属于当前用户，属于真实数据边界错误，而不只是命名不雅。

**修复建议**

落实既有设计中的单用户访问模型：使用配置的稳定用户 ID 和 `PROPULSE_ACCESS_TOKEN`，个人接口统一要求 bearer token；前端使用“解锁个人空间”状态，不把 token 写入静态文件或 URL。

**验收标准**

- 未解锁访问个人接口返回 `401`，页面展示明确的锁定状态，不展示样例。
- 生产代码、迁移和 OpenAPI 描述中不再出现 `demo-user` 或“demo user's”。
- 测算、观察池和出手窗口使用同一配置身份，并有未授权测试。

### SYS-002 [P0] Compose 默认把演示小区和指标写入业务库

**证据**

`docker-compose.yml` 为 API、worker 和 scheduler 设置 `PROPULSE_SEED_DEMO_DATA: "true"`；`SeedDemoData` 会写入“青枫花园”“云澜府”及整套指标。

**问题与影响**

通过正常启动方式得到的数据库会混入看似真实的市场记录。后续接入真实导入后，演示记录仍可能出现在观察池、出手建议和定时计算中，无法仅靠前端识别。

**修复建议**

把 seed 放到显式 demo profile 或独立命令，默认集成环境不 seed；演示数据需要独立数据集/环境和显著的“演示模式”标识，不能与真实数据同库混用。

**验收标准**

- 默认 `docker compose up` 不创建演示小区。
- 只有显式启用 demo profile 才能 seed，重复运行保持幂等。
- 真实模式的 API 响应中不会出现 seed 记录。

### SYS-003 [P0] 缺少可信真实数据导入闭环

**证据**

当前只接受 `sourceType = manual_json` 的管理接口；请求只有自由文本 `sourceRef` 和简化记录。没有数据源实体、CSV、稳定房源 ID、覆盖范围、校验报告、批次详情或前端数据管理页。scheduler 只重算已有快照，不采集新数据。

**问题与影响**

即使删除前端假数据，也没有足够的流程把来源明确、可复核的真实市场数据持续送到页面。“定时更新”会重复计算旧批次，不能让数据变新。

**修复建议**

先完成可信手工数据闭环：数据源、导入批次、JSON/CSV、稳定来源记录 ID、采集时间、完整/部分覆盖、逐行校验和批次详情。自动采集可以后续实现，但不能用重算任务替代采集。

**验收标准**

- 管理员可选择真实数据源和小区导入 JSON/CSV，并查看校验与结果。
- 每个观察记录都能追溯到来源、批次和采集时间。
- scheduler 明确区分“采集新数据”和“重算已有数据”；没有新批次时不伪造更新时间。

### SYS-004 [P0] 缺少支撑周变化和趋势的时间序列指标

**证据**

- 当前 API 仅提供 latest metric，没有历史查询。
- `ListedHomesChangePct` 存在于应用类型中，但不在 `neighborhood_metrics` schema、写入 SQL 或 metric service 结果中，真实导入后通常为 0。
- 每小时 scheduler 会基于同一个最新 collection run 再插入一份 metric，可能产生内容相同但时间更新的伪历史。

**问题与影响**

“较上月”“本周变化”“连续 4 周”“近 8 周”等语义无法从当前存储可靠计算；重复重算还可能制造数据更新的假象。

**修复建议**

以 collection run 的采集时间为时间轴，保存批次关联和可比较指标；提供 8 周历史与周/月变化 API；同一输入批次的重复计算需要幂等或明确标识为重算。

**验收标准**

- 每个 delta 都能指出基准批次和当前批次。
- 历史 API 返回真实采集时间、缺口和覆盖范围。
- 没有历史时显示“暂无可比数据”，不显示 0%、`(-)` 或虚构趋势。

### SYS-005 [P0] “成交强弱”使用了无关代理公式

**证据**

`internal/application/metric/calculate_neighborhood.go` 的 `calculateMomentum` 只看 `listedHomes` 和 `priceCutHomes / listedHomes`：挂牌不少于 40 且降价占比不低于 20% 就判为成交弱，挂牌少于 20 且降价占比低于 10% 就判为成交强。它没有使用成交数量、成交日期或历史成交变化。

**问题与影响**

页面把这个结果显示为“成交偏弱/平稳/活跃”，并进一步影响小区状态和出手建议。指标名称和计算事实不一致，会产生实质性错误决策。

**修复建议**

引入独立成交观察数据，按明确窗口计算成交量、成交频率和变化；如果数据不足，返回 `unknown/insufficient`，不要用供应和降价数据冒充成交数据。

**验收标准**

- 成交动量仅由有来源的成交数据计算，并记录样本数和时间窗口。
- 无成交样本时页面显示“成交数据不足”。
- 信号与行动建议能处理未知状态，不自动把未知当作平稳或偏弱。

## 6. 首页问题

### HOME-001 [P1] 决策仪表盘样例没有就近标注

**具体假数据**

- 换房能力：安全总价 520 万、勉强总价 580 万、危险总价 620 万以上、月供偏安全。
- 小区：青枫花园、房东预期松动、挂牌连续增加、降价房源变多、成交量偏弱、议价空间扩大。
- 建议：可以开始看房但不急下定，重点关注 500-530 万三房和挂牌超过 60 天房源。

**证据**

这些值直接写在 `apps/web/src/components/home-page.tsx`。卡片标题只有“决策仪表盘 预览”，没有说明人物、小区和数字都是虚构示例。页脚的全站“Prototype”不足以给具体数字提供来源标识。

**修复建议**

首页可以保留场景，但卡片标题、数字区域和建议区域都应明确标注“示例数据/虚构场景”，或在用户解锁后改为读取其真实摘要。

**验收标准**

- 未使用真实 API 时，第一视口内能明确看出这些数字不是用户数据。
- 屏幕阅读器也能读到示例标识。
- 首页样例不被复制到业务页作为失败回退。

### HOME-002 [P0] “只看真实数据/数据驱动”与当前实现不符

**证据**

首页写有“不看营销软文，只看真实数据”，页脚写有“高端数据驱动”；但小区页完全静态，观察池和出手窗口会静默回退到 sample data，真实数据导入也缺少质量元数据。

**问题与影响**

这是对数据可信度的直接承诺。当前实现无法满足，会放大所有假数据的误导风险。

**修复建议**

在 SYS-003 至 SYS-005、NEI-001 和 WATCH-001 完成前，改成准确的原型/示例说明；完成后再以来源、时间、覆盖率和质量提示支撑“真实数据”承诺。

**验收标准**

- 产品文案与当前运行模式一致。
- 声称真实的每个市场结论都能查看来源和更新时间。

### HOME-003 [P1] “添加目标小区”不是添加流程

**证据**

首页按钮只链接 `/neighborhoods`，该页立即展示固定的青枫花园，没有搜索、选择、创建或确认步骤。

**修复建议**

实现城市/板块/小区搜索与目标户型选择，再调用创建小区和加入观察池接口；在该流程完成前，把按钮改为准确的“查看示例小区”或暂时移除。

**验收标准**

- 用户可从首页添加一个真实目标小区并在观察池中看到它。
- 取消或失败不会偷偷加入默认小区。

## 7. 换房测算问题

### CALC-001 [P1] 表单默认填入虚构家庭财务数据

**具体假数据**

`defaultHousingInput` 默认包含：现金 150 万、旧房 320 万、剩余贷款 80 万、月收入 3.5 万、现有月供 0、可接受月供 1.5 万、目标总价 550 万、装修 40 万、交易成本 18 万、过渡租金 5 万。

**问题与影响**

页面加载后立即用这套数据生成“你的”诊断，用户可能在未输入完整信息时把样例结果当作个人结果。

**修复建议**

使用空表单和未计算状态；如果需要“一键载入示例”，必须由用户主动触发并显示持续的示例标签。

**验收标准**

- 首次进入不展示个人诊断数字。
- 必填项完整且通过校验后才允许生成报告。
- 示例场景不会被保存为用户最近一次测算，除非用户明确确认。

### CALC-002 [P0] 两个参与计算的字段被隐藏并固定

**证据**

`currentMonthlyMortgage = 0` 和 `transactionCosts = 18` 会进入前后端计算，但 `fields` 中没有对应输入框。用户无法看到或修改它们。

**问题与影响**

结果看似基于用户输入，实际混入隐藏样例。尤其现有月供和税费会直接影响现金流与资金缺口。

**修复建议**

把所有计算入参显示为可编辑字段；可由政策/估值推导的成本必须显示公式、来源和允许覆盖，不能静默固定。

**验收标准**

- UI 展示并提交所有十个计算字段，或 API 删除不再需要的字段。
- 报告可展开查看最终入参与单位。
- 测试覆盖现有月供和交易成本变化对结果的影响。

### CALC-003 [P0] 550 万触发固定结果覆盖

**证据**

只要 `targetTotalPrice === 550`，页面就把安全总价显示为 520 万、首付缺口显示为约 35 万、月供收入比显示为 42%，即使用户已经修改现金、收入、旧房价值或其他字段。

**问题与影响**

这是明确的结果伪造条件，同一组输入在前端展示和实际计算对象中会产生不同数值。

**修复建议**

删除 `isReferenceScenario` 和全部 display override，所有结果只使用一次真实计算的返回值。

**验收标准**

- 任意字段变化都会按同一规则更新三个结果。
- 550 万与 549/551 万没有特殊代码路径。
- 回归测试明确禁止参考场景覆盖。

### CALC-004 [P0] 前后端双重计算且失败时继续展示本地结论

**证据**

前端始终调用 `calculateHousingCapacity` 生成完整结果；后端成功时只覆盖 `pressureLevel` 和 `strategy`，其余数字仍来自前端；后端失败时测试明确要求“保留本地预览”。页面仍写“基于最新数据生成”。

**问题与影响**

两套规则可能漂移，报告会混合两个版本的结果。接口失败只显示一行小错误，但核心数字和建议仍像已验证结论。

**修复建议**

由 Go 返回完整结果、原因、计算记录 ID、版本和时间。前端本地逻辑最多用于明确标注的草稿预览，不能与正式报告合并。

**验收标准**

- 正式报告的每个字段来自同一个 API response。
- 请求中、失败、成功状态清晰区分；失败不显示“最新报告”。
- 前后端不再维护两套生产计算规则。

### CALC-005 [P0] 诊断说明包含固定个人结论

**具体假数据**

- “旧房回款占比超 60%”。
- 非危险状态一律写“首付资金存在约 35 万缺口”。
- 一律建议“降低目标总价至 520 万以内”。
- 压力指针只会落在 18%、45%、82% 三个固定位置。

**问题与影响**

用户修改输入后，说明文字和可视位置可能与实际 `downPaymentGap`、`safeTotalPrice` 及比例冲突。

**修复建议**

让后端返回结构化 reasons 和实际边界值；前端只做格式化，不在 JSX 中嵌入个人数字或固定结论。

**验收标准**

- 文案里的金额、比例和风险等级都能追溯到本次 response。
- 0 缺口时不再出现“约 35 万缺口”。
- 指针按连续值或清楚的分档规则定位，并与文字一致。

### CALC-006 [P1] 财务假设不可见、不可配置、不可追溯

**证据**

前后端都硬编码 35% 首付、0.0041 月供系数、6 个月现金储备、35%/45%/55% 压力线等假设。页面没有贷款利率、期限、还款方式、城市政策或假设日期。

**问题与影响**

这些不是用户真实数据，也不是普适常量。政策或利率变化后，页面仍会给出带“精准计算”语气的结果。

**修复建议**

把规则版本、政策适用范围和默认值纳入后端配置；让用户确认或编辑关键贷款参数，并在报告中展示假设。

**验收标准**

- 报告包含利率、期限、首付比例、还款方式、规则版本和生效日期。
- 默认值有来源说明，修改后结果可复现。

### CALC-007 [P2] 方法卡片是假链接

**证据**

“为什么月供安全线比总价更重要？”和“旧房迟迟卖不掉怎么办？”使用带 `cursor-pointer` 的 `<article>`，没有链接、按钮或事件。

**修复建议与验收标准**

接到对应方法文章并支持键盘访问；未有内容时去掉可点击样式。点击、回车和辅助技术读出的语义必须一致。

## 8. 目标小区问题

### NEI-001 [P0] 页面没有读取小区 API

**证据**

`NeighborhoodsPage` 不调用 `get neighborhood` 或 `get metrics`，也不读取 `?id=`。组件虽接受 `input`，实际只有小区名称使用 `input.name`，其余展示仍是固定常量。

**问题与影响**

即使数据库已有真实小区和指标，该页也永远展示同一个样例市场。

**修复建议**

通过 `?id=<uuid>` 明确选择小区，分别加载基础信息、最新指标与历史；实现 loading、not found、no metric、stale、error 状态。

**验收标准**

- 两个不同小区 ID 会展示各自的名称、区域、户型、指标和状态。
- 无 ID 时进入小区选择/空态，而不是默认青枫花园。
- 请求失败不展示任何样例指标。

### NEI-002 [P0] 小区身份、时间和当前指标均为固定值

**具体假数据**

- 身份：青枫花园、滨江核心区、三房 90-110㎡、更新时间“今天 10:30”。
- 当前挂牌价 520-620 万，较上月 -2%。
- 近 90 天成交价 495-545 万，价差约 8.5%。
- 在售 42 套，较上月 +18%。
- 本周降价 11 套，占比 26%。
- 平均成交周期 78 天，逐渐拉长。
- 带看转定率 2.4%，成交偏弱。

**问题与影响**

六项卡片全部来自 `metrics` 常量，其中“带看转定率”后端甚至没有字段；页面的“平均成交周期”也不能直接等同于后端现有的 `avgDaysOnMarket`（挂牌时长）。相对变化和“今天”会随日期继续显示，形成伪新鲜度。

**修复建议**

只显示 API 有来源的字段；时间使用真实 `collectedAt/calculatedAt` 绝对时间和相对新鲜度；无数据的“带看转定率”等字段先删除，直到有可靠来源。

**验收标准**

- 每项指标包含统计窗口、样本数、来源和更新时间。
- 过期数据有明显警告，不能继续显示“今天”。
- API 无字段时 UI 不自行补造数字。
- 挂牌时长与成交周期使用各自真实数据和准确名称，不混用概念。

### NEI-003 [P0] 近 8 周趋势图是手写数组

**具体假数据**

挂牌柱值固定为 `[20, 22, 25, 24, 30, 35, 38, 42]`；横轴固定为 `W1` 至 `W8`；降价点位用 `index * 2 + 5` 生成，并不代表任何降价数据。

**问题与影响**

图表标题和图例把纯视觉造型表述成真实挂牌量和降价频次趋势。

**修复建议**

接入 SYS-004 的历史 API，横轴使用真实周/日期；缺周、部分覆盖和样本不足需要显式表示，不能插值伪造。

**验收标准**

- 图上每个点都能对应一个真实批次或明确聚合窗口。
- tooltip 显示日期、值、来源与覆盖状态。
- 少于两期数据时显示“暂无趋势”，不绘制示例图。

### NEI-004 [P0] 研判结论、依据和建议是固定文案

**具体假内容**

- “适合试探性砍价”“买方窗口开启”。
- “挂牌量连续 4 周增加”“房东预期开始松动”。
- 500-530 万、挂牌超过 60 天、砍价 5%-8%。
- 三条依据：供应显著增加、11 套降价、成交尚未放量。

**问题与影响**

这些文案不会随真实 API、组件 `input` 或数据质量改变，属于高风险的虚构行动建议。

**修复建议**

状态、reasons 和 advice 全部读取后端本次指标 response；金额范围来自真实成交区间/用户预算；数据不足时返回观察重点而非买卖结论。

**验收标准**

- 改变 API fixture 会同步改变结论、依据和建议。
- 每条 reason 能链接或定位到对应指标。
- stale、partial、insufficient 状态不会输出“窗口开启”。

### NEI-005 [P1] 提醒和加入观察池按钮没有行为

**证据**

“降价提醒”“加入观察池”都是无 `onClick` 的普通按钮。当前阶段也没有通知规则/投递能力。

**修复建议**

“加入观察池”接入已鉴权 API 并显示重复、成功和失败状态；在通知能力落地前删除“降价提醒”，或改成真实的页面内异常信号设置。

**验收标准**

- 每个可见按钮都有可验证结果和错误反馈。
- 页面不承诺尚不存在的通知投递。

## 9. 观察池问题

### WATCH-001 [P0] 首屏和错误状态静默展示虚构小区

**具体假数据**

- 青枫花园：滨江核心、三房、42 套在售（+8）、11 套降价、成交偏弱、适合砍价、建议约看 500-530 万三房。
- 云澜府：城东新区、四房、28 套在售、2 套降价、成交平稳、继续观察、超预算暂缓约看。

**证据**

组件初始 state 就是 `fallbackCommunities`；任意非 abort 错误都会再次设置同一 fallback。没有 loading 或 error 标识。

**问题与影响**

网络失败、服务故障和真实列表都可能在视觉上变成一份“正常的个人观察池”。

**修复建议**

初始值为空，显式实现 loading、locked、empty、success、stale 和 failed 状态；失败时提供重试，不得使用样例市场数据。

**验收标准**

- 首次请求完成前不显示小区卡片和汇总数字。
- 失败时只显示错误与重试，不出现青枫花园/云澜府。
- 测试删除“错误时保留 fallback”的期望。

### WATCH-002 [P0] 汇总数字矛盾且周变化语义不成立

**具体假数据与错误语义**

- fallback 只渲染 2 张小区卡，却显示关注小区 5 个、降价信号 2 个、可砍价 1 个、价格偏硬 2 个。
- API 成功后，只要某小区当前 `priceCutHomes > 0` 就计入“出现降价信号”，没有与上周比较。
- 标题写“本周变化”，但 live card 的 `listedDelta` 永远是空字符串。

**修复建议**

汇总必须由当前渲染集合和真实历史 delta 统一计算；把“当前有降价房源”和“本周新出现降价信号”区分为不同指标。

**验收标准**

- 汇总可由卡片逐项复算且总数一致。
- 没有历史基线时显示“暂无本周对比”，不把 snapshot 当 delta。
- 时间窗口在 API 和 UI 中有明确定义。

### WATCH-003 [P0] 异动提醒始终是固定事件

**具体假内容**

- “青枫花园新增 2 套目标户型，挂牌价处于低位。”
- “星河湾有 1 套房源单次降价 30 万。”

**证据**

只要 `communities.length > 0` 就渲染这两条；它们与 API 返回的小区无关，星河湾甚至不在 fallback 的两张卡片中。

**修复建议**

由相邻真实批次生成结构化异常事件，包含小区 ID、事件类型、前后值、发生时间和来源；未实现前显示“异常信号功能尚未有数据”或移除模块。

**验收标准**

- 接口只返回“接口花园”时页面不会出现青枫花园或星河湾。
- 每条事件可打开对应小区和证据。

### WATCH-004 [P1] 无指标小区被表示为正常的 0/空值

**证据**

后端无 metric 时返回状态“继续观察”、0 套在售、0 套降价和空的 `transactionMomentum`；Watchlist API 没有 `hasMetric`/数据状态，空 momentum 还违反 OpenAPI enum，前端映射后可能显示空白。

**问题与影响**

“没有采集数据”会被误解为“真实在售为 0”，并污染汇总统计。

**修复建议**

API 返回明确 `dataState`、nullable metrics、更新时间和质量信息；无数据卡片只显示待导入/待计算，不参与市场信号统计。

**验收标准**

- 无指标不会输出伪 0 或不符合 schema 的空枚举。
- 汇总明确区分有数据和待采集小区。

### WATCH-005 [P1] “导出本周报表”是假导出

**证据**

该链接的 `href` 是 `/templates`，不会生成、下载或复制任何本周数据。

**修复建议与验收标准**

实现带当前周次、数据来源和观察池快照的真实 CSV/PDF/可复制报表；在此之前将文案改为“查看报表模板”。点击“导出”必须产生可验证文件，失败需反馈。

### WATCH-006 [P1] 复盘记录无法保存

**证据**

两个输入没有受控 state，保存按钮没有 handler，后端也没有 review/note API。输入内容在刷新后丢失。

**修复建议与验收标准**

实现复盘数据模型、鉴权 CRUD、保存状态和历史查看；在能力完成前移除表单或明确标为不可保存的原型。保存成功后刷新页面仍能读取相同内容。

## 10. 出手窗口问题

### ACTION-001 [P0] 缺前置数据或请求失败时展示虚构建议

**具体假内容**

- 策略：“积极看房，大胆砍价”；信心：“中高（75%）”。
- 假定目标为青枫花园、预算偏安全、买方窗口初期。
- 三条行动包含 500-530 万、2 套挂牌超过 60 天、砍 5%-8%、旧房每周少于 2 组则降 2%。
- 三条风险包含旧房低于 310 万、成交连续两周放大等假设。

**证据**

组件初始 state 就是 fallback；`capacity_required` 和所有其他请求错误也都使用它。缺观察池、缺 metric、401 和 500 没有分别处理。

**问题与影响**

最需要阻止建议的场景反而得到一条非常具体的“砍价”建议，并且 75% 不是 API 支持的置信度格式。

**修复建议**

删除 runtime fallback；按 locked、capacity required、watchlist required、metric required、stale/insufficient 和 server failure 分流，提供对应下一步入口。

**验收标准**

- 任一前置条件缺失都不显示行动、信心、清单或风险。
- API 失败时页面不出现 75%、青枫花园或固定金额。
- 各错误码有独立 UI 测试。

### ACTION-002 [P0] 决策因子矩阵始终是固定样例

**六项固定内容**

1. 预算偏安全，首付充裕、月供可控。
2. 旧房回款需观察，旧房挂牌中、看房量一般。
3. 目标小区明显松动，降价房源连续三周增加。
4. 成交偏弱，买家话语权高。
5. 户型稀缺度中等，同户型约 12 套。
6. 有备选，已添加 2 个同板块替代小区。

**问题与影响**

即使 API 返回“建议出手/等待”，矩阵仍显示以上砍价场景，页面内部会互相矛盾。

**修复建议**

扩展后端 response，让全部 factor、状态、证据、数据时间和质量来自同一次决策；前端不保留 `matrix` 常量。

**验收标准**

- API fixture 改变时矩阵和核心建议同步变化。
- 每个因子可追溯到测算记录或小区指标。
- 不完整证据会降低/取消建议，而不是补固定文案。

### ACTION-003 [P1] 用户不能明确选择本次判断的小区

**证据**

前端调用 action-window 时不传 `neighborhoodId`；后端默认选观察池第一条。页面也不显示稳定 ID 或选择器。

**问题与影响**

列表顺序变化就可能改变建议对象，用户无法确认“当前建议是针对哪个小区”。

**修复建议与验收标准**

要求显式选择已关注小区，并将 ID 传给 API；页面标题和证据区显示小区名称、户型和数据时间。刷新后选择稳定，目标不可用时要求重新选择。

### ACTION-004 [P0] “更优备选”只按数量判断

**证据**

服务把 `AlternativesBetter` 赋值为 `len(watchlist) > 1`，没有比较价格、预算、信号、户型或质量；领域逻辑据此把砍价置信度从中提高为高。

**问题与影响**

“有第二个小区”不等于“存在更优备选”，却会抬高决策信心。

**修复建议与验收标准**

定义可解释的替代小区比较维度并返回比较证据；在比较功能完成前不要使用 `alternativesBetter` 提升置信度。只有满足明确规则且数据质量足够时才能标记为更优。

### ACTION-005 [P2] 行动清单的方框不可交互

**证据**

行动项前渲染的是带边框的 `<span>`，视觉上像复选框，但不能勾选、不能键盘操作，也不保存进度。

**修复建议与验收标准**

如果只是项目符号，改成无交互的清晰图标；如果要跟踪行动，使用真实 checkbox 并持久化完成状态。视觉语义、DOM 语义和实际行为必须一致。

## 11. 判断方法问题

### METHOD-001 [P1] 六个目录项只有一篇内容

**证据**

六个链接全部是 `href="#main-method"`，页面始终显示“挂牌变多但成交弱，说明什么？”，只有第一项显示 active。其他五个标题只是目录占位。

**修复建议与验收标准**

为每个主题提供唯一 slug/anchor 和对应文章，或只保留已完成主题。点击每项后标题、正文、URL/anchor 和 active 状态都必须匹配，并支持直接打开链接。

### METHOD-002 [P2] 方法阈值没有来源或适用范围

**证据**

页面把“连续四周”“降价占比超过 20%”“在成交底价下砍 3%-5%”写成通用规则，没有城市、市场阶段、样本要求、更新时间或方法来源。

**问题与影响**

这些不是页面实时假数据，但属于未经说明的硬编码决策参数，容易被当成普适事实。

**修复建议与验收标准**

为方法规则提供版本、适用条件、局限和来源；与产品计算规则共享同一版本说明。无法验证的比例改为定性建议，不承诺固定幅度。

## 12. 工具模板问题

### TEMPLATE-001 [P1] “复制模板结构”没有复制内容

**证据**

六张卡片的数据来自 `sample-data.ts`，每张只列四个字段标签；六个按钮都没有 handler，却标记“可复制”。

**问题与影响**

用户拿不到表格、Markdown、CSV 或剪贴板内容，当前只是模板目录占位。

**修复建议与验收标准**

为每种模板定义完整、可版本化的结构和输出格式，复制按钮写入真实剪贴板内容并给出成功/失败反馈；也可提供 CSV/XLSX 下载。自动化测试验证剪贴板或下载内容，而不只验证标题存在。

## 13. 全站导航问题

### NAV-001 [P1] “登录”按钮没有登录/解锁行为

**证据**

顶部“登录”是没有事件的 `<button>`，同时个人接口又固定使用 `demo-user`。

**修复建议与验收标准**

按 SYS-001 实现“解锁个人空间”、已解锁状态和退出；未实现前移除按钮，不能用无行为的“登录”暗示账户体系存在。按钮需支持键盘、错误反馈和会话过期处理。

## 14. 建议实施顺序

1. **先止损**：HOME-001/002、CALC-003/005、NEI-001 至 004、WATCH-001/003、ACTION-001/002。先让样例有标签，让失败和缺数据诚实暴露。
2. **建立可信底座**：SYS-001 至 SYS-005，完成访问控制、演示隔离、可信导入、历史指标、成交指标和数据质量元数据。
3. **接通核心决策链**：CALC-002/004/006、WATCH-002/004、ACTION-003/004，保证测算、小区、观察池和出手窗口使用同一批真实证据。
4. **补齐操作闭环**：HOME-003、NEI-005、WATCH-005/006、TEMPLATE-001、NAV-001。
5. **清理占位体验**：CALC-007、ACTION-005、METHOD-001/002，并删除生产组件不再使用的 sample data 与旧 fallback 测试。

## 15. 全局完成定义

所有问题完成后，应满足以下共同条件：

- 生产页面中不再从 `apps/web/src/lib/sample-data.ts` 读取个人或市场数据；该文件仅可存在于测试或明确标注的静态示例中。
- API 请求失败、未授权、空数据、数据过期和样本不足都有不同状态，任何状态都不会静默变成演示数据。
- 每个市场数字至少带数据源、采集时间、统计窗口、样本数/覆盖状态；每个个人结论带测算记录和规则版本。
- 所有出现“本周、上月、连续、趋势、最新”的文案都能由可追溯的时间序列数据证明。
- 所有可见按钮、链接、复选框和“可复制/可导出/可保存/可提醒”承诺都有真实行为、失败反馈和自动化测试。
- 通过生产代码扫描：除测试、历史文档和显式 demo profile 外，不存在 `fallbackCommunities`、`fallbackActionWindow`、`demo-user`、固定“今天 10:30”或参考价格 display override。
