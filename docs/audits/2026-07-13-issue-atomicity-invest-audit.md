# Issue 原子性（INVEST）评审报告（2026-07-13）

## 1. 评审目的

对当前仓库全部 65 个 GitHub Issue（10 个 Epic + 55 个可执行叶子任务）逐项评估其
是否达到「原子级看板任务」标准，给出可复现的评分，并对不达标项提供可直接落地的
拆分/改写建议。

配套材料：

- 数据来源审计见 [假数据与占位功能审计问题清单](2026-07-13-fake-data-and-placeholder-issues.md)。
- 本报告只评估「Issue 作为看板卡是否可交付」，不评估功能本身的取舍。

## 2. 评分方法

采用 INVEST 六维加权，每维 0–5 分整数，加权到 0–100：

| 维度 | 含义 | 权重 |
| --- | --- | --- |
| Independent 独立性 | 能否不依赖未完成兄弟任务而推进，且与其它 Issue 无范围重叠 | 20 |
| Small 规模小 | 单一关注点、单个 PR、≤1–2 天 | 20 |
| Testable 可测试 | 验收标准具体、可观测，而非复述目标 | 20 |
| Valuable 有价值 | 交付可演示价值或可验证能力 | 15 |
| Estimable 可估算 | 范围/路径清晰到能估工作量 | 15 |
| Negotiable 可协商 | 描述结果、给实现留余地 | 10 |

评级规则（叶子任务）：

- **达标**：总分 ≥ 80 且 S、T、I 均 ≥ 3。
- **需拆分/重写**：总分 < 60，或 S ≤ 2，或 T ≤ 2。
- **需微调**：其余。

Epic 不套用原子量表结论（其 S/E/I 天然偏低），单独以「拆分完备度」判断，
统一标注 `Epic·拆分到位/基本可用/不足`。

评审由 11 个并行评估器按统一锚点打分，再经一个综合校准器复核跨批次一致性；
全部总分经独立重算校验，无算术误差。

## 3. 总体结论

**这套 Issue 的原子化程度整体较高**，尤其 30 个子任务（#36–65）几乎都是
「单一关注点 + 单 PR + 可测验收」的合格看板卡。真正需要动手的问题集中在
少数几张卡和几个**系统性模板缺陷**上，而非普遍的「拆得不够细」。

评级分布（55 个叶子任务）：

| 评级 | 数量 | 占比 |
| --- | --- | --- |
| 达标 | 17 | 31% |
| 需微调 | 34 | 62% |
| 需拆分/重写 | 4 | 7% |
| Epic（拆分到位） | 10 | — |

### 关键校正：62% 的「需微调」多数不是真的不原子

34 个「需微调」大多被压在 73–79 分，**不是因为任务太大或不可测，而是被两个
结构性因素封顶**，属正常现象、无需返工：

1. **独立性 I=3**：这些卡有真实上游依赖（`Blocked by`），在分层构建里是
   必然的，不是缺陷。一个正确排序的依赖 DAG 本就该如此。
2. **价值 V=3–4**：纯后端 / schema / client 环节，价值只在链条末端体现，
   无法独立演示。

结论：这批卡在规模(S)与可测性(T)上合格，分数上不去纯是依赖结构与加权的
天花板效应，**不需要重写**。

  改成可观测项，并把版本机制依赖显式列为 `Blocked by`。

## 5. 跨任务共性缺陷（比单卡更值得治）

1. **验收标准原样复述「目标 / 期望行为」** —— 影响 #15 / #25 / #26 / #29 / #30 /
   #33 / #35。模板字段被复制填充，是拉低 T 分、触发返工的头号原因；开发「做完」
   与「验收通过」脱钩。
2. **单卡塞多条互斥实现路径**（完整实现 vs 占位/删除）—— 影响 #7 / #23 / #25 /
   #32 / #34 / #35。路径未定 → 工作量在 0.5 天和数天间漂移，破坏看板可预测性。
   应在进入就绪态前二选一。
3. **相邻 Issue 范围交叠未划边界** —— #10 与 #14（假设来源展示）、#7 与 SYS 系列
   （数据承诺）。应在模板中点名归口方并互设 `Blocked by`。
4. **模板化打分暴露的同质化** —— 16 张子任务拿到完全相同的 INVEST 向量(76)、
   8 张相同(73)，说明「实现范围」退化成同一段样板文字。

## 6. Epic 拆分结论

10 个 Epic 子任务对父目标的覆盖均完整、互斥、成链，全部判「拆分到位」。
其中 #30、#26 需按 §4.2 补 Epic 级可观测验收。#16（NEI-001）子任务边界最干净，
可作为后续拆分参照。

## 7. 对 Issue 模板的系统性建议

1. **「目标」「期望行为」「验收标准」三字段禁止内容雷同**；验收强制写成
   Given/When/Then 或勾选式可观测断言，评审时验收与目标高度重合直接打回。
2. **建可复用验收片段库**（provenance 五要素、状态机 loading/empty/error/stale/
   insufficient、鉴权 401/匿名可读），避免实现范围退化成样板。
3. **「实现范围」禁止样板化**：要求列出具体文件 / 函数 / 迁移 / OpenAPI 路径，
   据此判断是否越过单 PR 上限。
4. **Epic 与原子任务分轨评估**：Epic 不进 0–100 原子量表，改用「拆分完备度」量表。

改进后的模板见 `.github/ISSUE_TEMPLATE/`。

## 8. 评分方法学的已知局限

- **N 维区分度低**：65 张中 62 张 N=4，且权重最低(10)，对排序几乎无贡献；
  后续应重新定义或从量表移除。
- **等级带曾有边界重叠**：本报告已用「总分 + S/T/I 门槛」双重规则消解，
  但仍有个案（如 #15）被规则判「需拆分/重写」而实际只需重写验收——已在 §4.1 标注。
- **子任务同质向量**：反映模板样板化，也可能掩盖子任务间真实的规模/可测性差异。
